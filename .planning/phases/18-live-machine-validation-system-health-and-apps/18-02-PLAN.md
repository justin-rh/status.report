<task type="checkpoint:human-action" gate="blocking">
  <name>Task 1: Edgar runs SCRY on enrolled Windows machines and populates `18-VALIDATION-RESULTS.md`</name>
  <what-built>
    Plan 18-01 created the `18-VALIDATION-RESULTS.md` skeleton with SC2 already marked PASS (Justin's pre-validation of both UPTIME_WARN and UPTIME_STALE badge states). The Edgar runsheet is embedded in the file. This checkpoint hands off to Edgar/IT to:

    **1. Obtain the SCRY executable:**
       - Get the SCRY `.exe` from `dist/scry/` on the current branch (ask Justin for the built copy on a flash drive if needed).
       - Copy the full `dist/scry/` folder to a USB flash drive — do NOT copy to the host PC's disk.

    **2. Run SC1 — Uptime + pending updates (Admin account):**
       - On an enrolled Windows machine, open Command Prompt or PowerShell as Administrator.
       - From the flash drive: run `scry.exe`
       - Open the generated `.html` file in Edge or Chrome.
       - Look for the SYSTEM / Health section: confirm uptime shows a real value (e.g. "9 days 4 hours", not "N/A") and pending Windows Updates shows a number (not "N/A").
       - Fill in `## SC1 — Uptime + Pending Updates (Admin Account)` in `18-VALIDATION-RESULTS.md`.

    **3. Run SC3 — Pending updates as standard user:**
       - On the same (or a different) enrolled machine, open Command Prompt or PowerShell as a standard user (do NOT run as administrator).
       - From the flash drive: run `scry.exe`
       - Open the generated `.html` file.
       - Look for **Pending Windows Updates** — it must show `N/A` (the WUA COM call requires elevation and degrades gracefully). Uptime may still show a real value; that is correct.
       - Fill in `## SC3 — Pending Updates as Standard User`.

    **4. Run SC4 — App detection + M365 sign-off:**
       - On an enrolled Windows machine (ideally Intune-enrolled for Company Portal), run `scry.exe` from the flash drive.
       - Open the generated `.html` file.
       - In the Equipment table, look for:
         - NinjaOne / NinjaRMM Agent — should appear as Installed.
         - CrowdStrike Falcon (CrowdStrike Windows Sensor or CrowdStrike Sensor Platform) — should appear as Installed.
         - Microsoft 365 — should appear as a **single row** (e.g. "Microsoft 365 — Word, Excel, Outlook, ..."), NOT individual rows per app.
         - Company Portal — on an Intune-enrolled machine, should appear as Installed.
       - For M365: write a brief text sign-off confirming the single-suite-entry display is acceptable (e.g. "Microsoft 365 appears as a single row with sub-apps listed; acceptable for IT purposes"). No screenshot needed (D-08).
       - If no Intune-enrolled machine is reachable, mark Company Portal as DEFERRED with rationale in the SC4 section.
       - Fill in `## SC4 — App Detection + M365 Sign-off`.

    **5. Run SC5 — HTML character sheet renders in real browser:**
       - Using the `.html` file from any of the above runs, open it in Microsoft Edge or Google Chrome.
       - Confirm: dark color scheme present, layout correct (stat block, equipment table, quest status section), stat block has real values, equipment table is styled (not broken), quest status section appears.
       - Fill in `## SC5 — HTML Character Sheet Render`.

    **6. Update the Summary table:**
       - At the top of `18-VALIDATION-RESULTS.md`, update the Summary table. Replace each `_pending Edgar run_` with the actual disposition (PASS / FAIL / DEFERRED). SC2 is already PASS — do NOT change it.

    **7. Commit `18-VALIDATION-RESULTS.md`** with all SC sections filled in. The commit must be separate from Plan 18-01's skeleton commit (this provides a clean audit trail).

    **DEFERRED path (D-04 — phase closes atomically):**
    If a SC cannot be validated in this run (e.g. no Intune-enrolled machine reachable for Company Portal, or Edgar unavailable for a specific scenario), mark that SC as DEFERRED with a documented rationale. The phase can close atomically when all SCs are PASS or DEFERRED. A DEFERRED SC does NOT block phase close — missing rationale does.

    **FAIL path:**
    If any SC produces unexpected behavior (e.g. uptime shows "N/A" under Admin, or CrowdStrike not detected on an enrolled machine), mark it FAIL, describe the exact symptom in `Divergence notes`, and provide that symptom in the resume signal. Plan 18-03 will add a conditional fix plan.

    This is the ONE point in Phase 18 where Claude cannot proceed autonomously — running SCRY on enrolled fleet hardware requires Edgar's access. There is no CLI or API substitute. However, a single Admin-account run + a single standard-user run + an Intune-enrolled machine run is sufficient to cover all four SCs.
  </what-built>
