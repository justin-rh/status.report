## Overview

Surfaced system health signals (uptime, pending Windows updates), vendor update detection (Dell Command Update, Lenovo System Update), and extended CLI output options (`--json`, `--output`, `--app`) so IT staff can assess machine state and integrate output into NinjaOne workflows. Also delivered the SCRY rebrand (StatusReport → SCRY) and added a `Warning.level` field powering yellow/red severity badges. Delivered 70 commits since v2.0, 29 source files changed (+1,629/-85 LOC excl. `.planning`); 291 tests passing at close (88 new, +43%).

### Phase 15: Extended CLI Flags

**Goal**: IT staff and NinjaOne scripts can retrieve audit output as JSON, override the output path, and query a single app — without generating a full HTML report when not needed
**Depends on**: Phase 14
**Requirements**: OUT-V3-01, OUT-V3-02, CLI-V3-01
**Plans**: 1 plan

Plans:
- [x] 15-01-PLAN.md — --json, --output, --app flags + tests

**Details:**
- `--json` writes JSON file alongside HTML in `logs_dir` (or `--output PATH`); uses `dataclasses.asdict(report)` + `json.dumps(..., default=str)` — recurses through nested `VendorUpdateStatus`, `AppStatus`, `Warning`, `ParsedHostname` dataclasses
- `--output PATH` overrides default `logs_dir`; any writable path accepted (no host-path validation per D-02/D-03)
- `--app NAME` runs single-app detection only, prints to stdout, exits with code 0 before any file write
- `--app NAME --json` prints JSON blob of single AppStatus to stdout, exits with code 0
- `_find_app_spec()` uses case-insensitive substring match: `q in spec["name"].lower()`
- Unknown app: stderr "Unknown app: NAME", exits with code 1
- `--json` combined with `--name`/`--serial`/`--warnings` forces full pipeline (`if cli_mode and not args.json:` guard)
- `_run_cli_app()` early-exits before HTML write logic — `--app NAME --output PATH` silently discards `--output` (per D-08/D-13)
- 7 new tests (284 → 291); zero regressions

## Milestone Summary

**Key Decisions:**

- Warning.level field is positional-LAST to preserve backward-compat with `Warning(code, severity, message, detail)` callers
- `_WIN32COM_AVAILABLE` guard pattern mirrors `_WMI_AVAILABLE`/`_PWD_AVAILABLE` — standard CI-compatible Windows COM guard, named after library not feature for future reuse
- Vendor detection is registry+file-only — vendor CLIs (`dcu-cli.exe`, `tvsu.exe`) explicitly prohibited (admin elevation, side effects, violate PKG-02)
- LSU pending count returns N/A by design — no admin-safe passive source available in v3.0; deferred to LSU-PENDING future requirement
- `--output PATH` accepts any writable path — host-path validation removed (D-02/D-03) per user decision; runs from a USB drive plugged into a host, "host PC" can be a write target if user chooses
- `--json` is an output-format modifier, not a pipeline mode selector — full pipeline always runs unless `--app` short-circuits
- DCU XML staleness shows "Unknown (no scan data)" not 0 — distinguishes "no data" from "0 pending"
- SCRY rename preserved git history via mechanical renames; one intentional historical parenthetical in CLAUDE.md retained for context
- Output filename format `{date}_scry_{hostname}.html` is date-first for alphabetical sort (audit-trail readability)

**Issues Resolved:**

- StatusReport → SCRY full rename without breaking 203 existing tests
- Warning system extended to yellow/red severity levels without breaking existing warning callers (level defaults None)
- WUA COM access fails gracefully under standard user (pending_updates → None → "N/A") without crashing
- `--json` + nested-dataclass serialization handled correctly by `dataclasses.asdict()` recursion
- Argparse + `--json` + targeted flags compose without conflict (full pipeline forced when JSON requested)

**Issues Deferred:**

- **11 hardware-gated human verification items**: live SYSTEM/Admin uptime + pending update count; uptime > 7d yellow badge; uptime > 30d red badge with hibernation note; standard-user N/A degradation; live Dell DCU count; live non-Dell/non-Lenovo "Not installed"; 3 visual HTML render checks for vendor row states
- **Dell Command Update registry path uncertainty**: documented as IT confirmation gate; code uses Phase 14 researched path
- **Lenovo System Update registry path uncertainty**: same status
- **LSU-PENDING**: no passive source; requires `tvsu.exe` (admin-only); deferred to future milestone if business need arises

**Technical Debt Incurred:**

- `writers.write_html` (writers/__init__.py:16) is dead code from the user pipeline — `main.py` calls `render_html()` directly. Hard-coded "scry.html" filename in dead path cannot affect output but is misleading. Recommend deprecation in v3.1
- `_run_cli` mode (`--name`/`--serial`/`--warnings`) honors `--updates` and runs vendor collection, but the data is never used — wasted work, not a bug
- `--app NAME --output PATH` silently discards `--output` (matches D-08/D-13 spec; not surfaced to user)
- REQUIREMENTS.md checkbox bookkeeping lag — for the third milestone in a row (v1.0, v2.0, v3.0) checkboxes were not kept in sync with execution; resolved at close. Worth automating before v4.0
